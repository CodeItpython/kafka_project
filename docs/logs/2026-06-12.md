# 2026-06-12 작업 로그

## Lombok과 어노테이션 기반 리팩토링

- `auth-service`에 Lombok 의존성을 추가했다.
- JPA 엔티티와 MongoDB/Elasticsearch 문서 클래스에 `@Getter`, `@NoArgsConstructor(access = PROTECTED)`를 적용했다.
- 의미 있는 상태 변경 메서드는 유지하고 반복 getter와 보호 기본 생성자를 제거했다.
- Spring Bean 일부에 `@RequiredArgsConstructor`를 적용해 생성자 주입 반복 코드를 줄였다.
- 로그 선언은 `@Slf4j`로 정리했다.
- 컨트롤러에 `@Validated`를 추가해 request parameter/path variable 검증 확장에 대비했다.

## 에러 응답과 에러 화면 템플릿

- API 에러 응답을 `ErrorResponse` record로 표준화했다.
- 응답 필드는 `timestamp`, `status`, `code`, `message`, `path`, `traceId`, `details`로 구성했다.
- `ApiExceptionHandler`가 검증 오류, 잘못된 요청, 서버 오류를 같은 형태로 반환하도록 변경했다.
- 서버 로그는 `api_error traceId/status/code/path/message` 형식으로 남기도록 정리했다.
- OAuth 콜백처럼 HTML 응답이 필요한 에러는 `ErrorPageTemplate` 컴포넌트로 분리했다.

## 로그인 UI와 로컬 실행

- 로그인 화면의 기술 설명 문구에서 JWT 같은 내부 구현 용어를 제거했다.
- 프론트 API 에러 처리에 `ApiClientError`를 추가해 서버 표준 에러 응답을 사용자 친화 문구로 변환하도록 변경했다.
- 로그인 실패 시 원시 에러 로그처럼 보이는 문구 대신 이메일/비밀번호 확인 안내가 보이도록 했다.
- 로컬 기본값은 `EUREKA_CLIENT_ENABLED=false`로 변경했다.
- Docker Compose와 Kubernetes에서는 `EUREKA_CLIENT_ENABLED=true`를 명시해 discovery-service가 있을 때만 Eureka 등록을 수행하도록 분리했다.

## 사용된 개념

- Lombok: 컴파일 타임 annotation processing으로 생성자, getter, logger 같은 반복 코드를 줄이는 도구다.
- Constructor injection: Spring Bean의 필수 의존성을 생성자에서 받도록 해 불변성과 테스트 용이성을 높이는 방식이다.
- Standard error response: 프론트가 에러를 안정적으로 처리할 수 있도록 서버 에러 응답의 필드와 code를 고정하는 방식이다.
- Trace ID: 서버 로그와 클라이언트 에러 응답을 연결하기 위한 식별자다.

## 로컬 실행 오류 보정

- 로컬에서 Logstash가 실행 중이 아닐 때도 `LogstashTcpSocketAppender`가 계속 연결을 재시도하며 `Broken pipe` 경고를 남기는 문제를 확인했다.
- `logback-spring.xml`을 Spring profile 기반으로 변경해 기본 로컬 실행에서는 콘솔 로그만 사용하고, `logstash` profile에서만 Logstash appender를 사용하도록 수정했다.
- Docker Compose와 Kubernetes의 auth-service에는 `SPRING_PROFILES_ACTIVE=logstash`를 명시해 컨테이너 배포에서는 ELK 로그 수집을 유지했다.
- Redis가 실행 중이 아닐 때 채팅방 목록 캐시가 실패하며 긴 stack trace를 출력하던 로그를 짧은 fallback 경고와 debug 상세 로그로 분리했다.
- `POST /api/chat/rooms/{roomId}/typing` 호출 시 트랜잭션 밖에서 JPA lazy collection을 읽어 `LazyInitializationException`이 발생하던 문제를 `@Transactional(readOnly = true)`로 보정했다.
- Spring Data Redis repository 자동 스캔은 사용하지 않도록 `spring.data.redis.repositories.enabled=false`를 추가했다.
- JPA, MongoDB, Elasticsearch repository 스캔 범위를 명시하는 `RepositoryConfig`를 추가했다.
- JPA/Mongo repository 스캔 필터와 Reactive Elasticsearch repository 자동설정 제외를 추가해 여러 Spring Data 모듈이 서로의 repository를 후보로 오해하는 로그를 줄였다.
- 테스트용으로 수정된 백엔드를 `8891` 포트에 실행해 로그인, 채팅방 목록 조회, typing API를 직접 호출했다.
- `POST /api/chat/rooms/{roomId}/typing` 응답이 204로 성공하고 더 이상 500이 발생하지 않는 것을 확인했다.

- IntelliJ에서 바로 실행할 수 있도록 로컬 `.idea/runConfigurations/AuthServiceApplication.xml`에 `--eureka.client.enabled=false` 실행 인자를 설정했다.

- `kafka-chat-ci`와 `kafka-chat-local-deploy`의 기본 감시 브랜치를 `*/codex/merge-redis-elk-lombok-to-main`로 변경했다.
- `kafka-chat-main-ci` job을 추가해 main 브랜치는 배포 없이 검증만 수행하도록 분리했다.
- 기능 브랜치를 전부 자동 배포하지 않고, 통합 테스트 브랜치만 로컬 Kubernetes 배포 대상으로 삼는 전략을 채택했다.
- Docker Compose의 Jenkins 서비스에 `KAFKA_PROJECT_GIT_URL`, `KAFKA_PROJECT_CI_BRANCH`, `KAFKA_PROJECT_MAIN_BRANCH`, `KAFKA_PROJECT_POLL_SCM` 환경변수를 추가해 브랜치 감시 대상을 명시적으로 바꿀 수 있게 했다.

- Redis 컨테이너를 로컬 Compose 구성으로 기동했다.
- Kibana와 Logstash는 이미지가 로컬에 없어서 `http://localhost:5601`이 열리지 않는 상태이며, 이미지 pull 속도가 비정상적으로 느려 완료하지 못했다.

## ELK 컨테이너 기동 보완

- Docker Desktop의 Kubernetes는 Docker Desktop 안에서 로컬 단일 노드 Kubernetes 클러스터를 켜는 기능이다. Kafka 컨테이너 안에 Kubernetes나 ELK를 설치한다는 의미가 아니다.
- Kafka, Elasticsearch, Logstash, Kibana는 각각 독립 컨테이너로 실행하고 같은 Docker Compose 네트워크에서 통신하도록 구성한다.
- Kafka 컨테이너는 메시지 브로커 역할만 담당하고, Logstash는 로그 수집/가공, Elasticsearch는 저장/검색, Kibana는 시각화 UI를 담당한다.
- macOS에서 5000번 포트가 AirPlay Receiver 등과 충돌할 수 있어 Logstash의 호스트 포트는 `localhost:5001`로 변경했다. 컨테이너 내부와 Docker Compose 네트워크에서는 계속 `logstash:5000`을 사용한다.
- `kafka-chat-ci #14`가 `codex/merge-redis-elk-lombok-to-main` 브랜치의 `Jenkins CI/CD 브랜치 전략 정리` 커밋을 기준으로 backend Gradle test와 frontend build를 성공했다.
- `kafka-chat-local-deploy #21`은 같은 브랜치 기준으로 backend/frontend 검증과 Docker 이미지 빌드까지 성공했지만, 로컬 kubeconfig에 current-context와 cluster가 없어 Kubernetes 배포 단계에서 실패했다.
- `Jenkinsfile.local-deploy`에 `Validate Local Kubernetes` 단계를 추가해 kube context가 없으면 Docker 이미지 빌드 전에 명확한 메시지로 실패하도록 보정했다.
