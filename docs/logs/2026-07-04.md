## 개념

- StatefulSet + volumeClaimTemplates: DB처럼 상태를 갖는 워크로드는 파드마다 안정적 식별자와 전용 PVC가 필요하다. Deployment는 볼륨을 명시하지 않으면 컨테이너 쓰기 계층에만 저장돼 재시작 시 데이터가 사라진다. StatefulSet은 `volumeClaimTemplates`로 파드별 PVC를 자동 생성하고, 거버닝용 헤드리스 서비스(`clusterIP: None`)로 `pod-0.<svc>.<ns>.svc` 형태의 안정 DNS를 제공한다. 애플리케이션은 기존 ClusterIP 서비스명으로 접속하므로 접속 설정 변경이 없다.
- 서비스 clusterIP 불변성: 이미 배포된 ClusterIP 서비스는 `clusterIP`를 나중에 `None`(헤드리스)으로 바꿀 수 없다(`may not change once set`). 그래서 기존 서비스는 그대로 두고 `-headless` 서비스를 별도로 추가하는 방식으로 in-place apply 충돌을 피했다. Deployment→StatefulSet은 kind가 달라 apply가 프루닝하지 않으므로, 배포 스크립트에서 구 Deployment를 먼저 삭제해 중복 파드를 막는다.
- 헬스체크와 기동 순서: compose의 `depends_on: condition: service_healthy`는 의존 서비스가 healthcheck를 통과한 뒤에야 앱을 기동한다. `start_period`는 초기 부팅 유예 구간으로, 이 시간 동안의 실패는 재시도 횟수에 포함되지 않는다. DB/Kafka 준비 전에 auth-service가 떠서 연결 실패로 재시작하는 문제를 없앤다.
- Spring Boot layered jar: bootJar 내부를 dependencies/spring-boot-loader/snapshot-dependencies/application 네 레이어로 분리해 추출(`-Djarmode=layertools ... extract`)한다. 이미지 레이어를 변동성이 낮은 순서로 COPY하면 코드만 바뀔 때 애플리케이션 레이어만 다시 빌드·전송되어 이미지 캐시 효율이 크게 오른다. 엔트리포인트는 `org.springframework.boot.loader.launch.JarLauncher`(Spring Boot 3.2+ 패키지).
- 컨테이너에서의 JVM 힙: `-XX:MaxRAMPercentage`는 컨테이너 메모리 한도(cgroup) 기준으로 힙 상한을 잡는다. 고정 `-Xmx` 대신 사용하면 k8s `resources.limits.memory`를 바꿔도 힙이 자동으로 따라가고 OOMKill 위험이 준다. 비root 실행(`USER spring`)은 컨테이너 탈취 시 피해를 줄이는 기본 보안 조치다.
- Kafka 프로듀서 durability: `acks=all`은 ISR 전체가 기록해야 성공 처리하고, `enable.idempotence=true`는 재시도로 인한 중복 기록을 막는다(프로듀서 ID + 시퀀스). 트랜잭셔널 아웃박스가 "발행 성공 = 내구성 보장"으로 동작하려면 프로듀서 자체가 durable해야 한다. idempotence는 브로커 통신 시점에만 관여하므로 브로커가 없으면 앱은 정상 기동하고 전송만 실패한다(기존과 동일).
- Graceful shutdown: `server.shutdown=graceful`은 SIGTERM 시 새 요청 수신을 멈추고 진행 중 요청을 마무리한다. K8s에서는 preStop 훅(엔드포인트에서 파드가 빠질 시간 확보)과 `terminationGracePeriodSeconds`를 함께 써야 롤링 업데이트 중 요청 드롭을 막는다.
- HPA + metrics-server: HorizontalPodAutoscaler는 파드 리소스 사용률 기준으로 레플리카 수를 조절한다. 사용률 계산에 metrics-server가 필요하며 Docker Desktop 단일 노드에서는 kubelet 인증서가 self-signed라 `--kubelet-insecure-tls` 플래그로 별도 설치해야 한다.
- 알려진 제약(후속 과제): WebSocket이 인메모리 `SimpleBroker`라 auth-service를 2 레플리카 이상으로 스케일하면 특정 인스턴스가 소비한 Kafka 메시지를 그 인스턴스의 로컬 세션에만 브로드캐스트한다 → 다른 인스턴스에 붙은 클라이언트는 실시간 수신을 놓친다. HPA를 실제 다중 레플리카로 활용하려면 공유 STOMP 브로커(Redis/RabbitMQ relay) 도입이 선행돼야 한다.
