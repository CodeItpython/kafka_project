## 개념

- StatefulSet + volumeClaimTemplates: 파드마다 안정적 식별자와 전용 PVC를 부여한다. DB처럼 상태를 갖는 워크로드는 Deployment(휘발성)가 아니라 StatefulSet으로 배포해야 재시작/재스케줄 후에도 데이터가 유지된다. 거버닝용 헤드리스 서비스(`<name>-headless`)로 안정 DNS를 제공하고, 클라이언트는 기존 ClusterIP 서비스명으로 접속한다.
- 헬스체크와 기동 순서: compose `depends_on: condition: service_healthy`는 의존 서비스가 healthcheck를 통과한 뒤에야 앱을 띄운다. DB/Kafka 준비 전 앱이 떠서 크래시-재시작하는 문제를 없앤다.
- Spring Boot layered jar: 의존성/로더/스냅샷/애플리케이션을 별도 레이어로 분리해 코드만 바뀌면 애플리케이션 레이어만 다시 빌드/전송한다. 비root 실행과 `-XX:MaxRAMPercentage`로 컨테이너 메모리 한도에 맞춰 힙을 잡는다.
- Kafka 프로듀서 durability: `acks=all`+`enable.idempotence=true`는 리더 페일오버 유실과 재시도 중복을 막는다. 트랜잭셔널 아웃박스가 "발행 성공=내구성 보장"으로 동작하려면 프로듀서 자체가 durable해야 한다.
- Graceful shutdown: SIGTERM 시 새 요청 수신을 멈추고 진행 중 요청을 마무리한다. K8s에서는 preStop(엔드포인트 제거 대기)+terminationGracePeriodSeconds와 함께 써야 롤링 업데이트 중 요청 드롭을 막는다.
- HPA + metrics-server: HorizontalPodAutoscaler는 CPU/메모리 사용률 기준으로 레플리카를 늘리고 줄인다. metrics-server가 있어야 동작하며 Docker Desktop에서는 `--kubelet-insecure-tls`로 별도 설치가 필요하다.
- 알려진 제약: WebSocket이 인메모리 SimpleBroker라 auth-service를 2개 이상으로 스케일하면 인스턴스 간 실시간 전달이 끊긴다. HPA를 실제 다중 레플리카로 쓰려면 공유 STOMP 브로커(Redis/RabbitMQ relay)가 선행돼야 한다.
