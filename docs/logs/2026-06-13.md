## Kafka Kubernetes 안정화

- Kafka KRaft 단일 노드 Pod가 CrashLoopBackOff로 반복 재시작되는 문제를 확인했다.
- 원인:
  - `kafka` Service가 broker 포트 9092만 노출하고 controller 포트 9093을 노출하지 않았다.
  - 단일 노드 Kafka를 RollingUpdate로 교체하면서 같은 `node.id=1` broker/controller가 잠깐 둘 이상 떠서 controller registration이 꼬였다.
  - Service가 Ready Pod만 endpoint로 잡으면 Kafka가 controller에 붙어야 Ready가 되는 순환 대기가 발생한다.
- 조치:
  - `kafka` Service에 controller 포트 9093을 추가했다.
  - Kafka Deployment 전략을 `Recreate`로 변경했다.
  - Service에 `publishNotReadyAddresses: true`를 추가했다.
  - Kafka Pod에 readiness/liveness probe와 resources requests/limits를 추가했다.
- 결과:
  - `scripts/deploy-local-k8s.sh`로 이미지 빌드, `kubectl apply`, auth/frontend/discovery rollout을 검증했다.
  - Kafka 수정 후 `kubectl -n kafka-project get pods` 기준 전체 Pod가 Running 상태가 됐다.

## 사용자 프로필과 상태메시지

- `users` 테이블에 현재 프로필 상태를 표현하는 필드를 추가했다.
  - `statusMessage`: 사용자의 현재 상태메시지
  - `profileImageUrl`: 현재 프로필 이미지 URL
  - `updatedAt`: 프로필 마지막 수정 시각
- `user_profile_history` 테이블을 추가해 프로필 변경 스냅샷을 저장한다.
  - 이름, 상태메시지, 프로필 이미지 URL, 이벤트 타입, 변경 시각을 남긴다.
  - 다른 사용자가 공개 프로필 화면에서 최근 변경 이력을 볼 수 있다.
- API를 추가했다.
  - `GET /api/users/me/profile`
  - `PATCH /api/users/me/profile`
  - `POST /api/users/me/profile-image`
  - `GET /api/users/{userId}/profile`
  - `GET /api/users/profile-images/{fileName}`
- 친구 목록 Redis 프로필 캐시에 상태메시지와 프로필 이미지 URL을 포함했다.
- 프론트엔드 사이드바에서 내 프로필 수정, 프로필 이미지 등록, 친구 공개 프로필/히스토리 조회, 공개 프로필에서 1:1 메시지 시작을 구현했다.
- 기존 PostgreSQL `users` 행이 있는 상태에서 `updated_at` 새 컬럼이 `NOT NULL`로 추가되면 애플리케이션 시작이 실패할 수 있어 nullable로 열고 애플리케이션 시작 후 백필하도록 보정했다.

## MinIO/S3 객체 스토리지 적용

- 채팅 첨부 이미지/GIF와 프로필 이미지를 저장하는 공통 `ObjectStorageService` 계층을 추가했다.
- 기본 로컬 IDE 실행은 `app.storage.type=local`로 유지한다.
  - 로컬 저장 루트: `uploads`
  - 채팅 첨부 키: `chat/{fileName}`
  - 프로필 이미지 키: `profiles/{fileName}`
- Docker Compose와 Kubernetes 배포에서는 `APP_STORAGE_TYPE=s3`로 전환해 MinIO를 S3 호환 객체 스토리지로 사용한다.
- 브라우저에 내려주는 URL은 기존처럼 백엔드 프록시 URL을 유지한다.
  - `/api/chat/attachments/{fileName}`
  - `/api/users/profile-images/{fileName}`
- 메시지 본문과 파일 메타데이터는 MongoDB에 남고, 검색 인덱스는 Elasticsearch가 맡는다. MinIO는 실제 바이너리 파일 본문만 저장한다.
- Kubernetes에 `minio` Deployment/Service를 추가했다.
  - API NodePort: `30900`
  - Console NodePort: `30901`
  - readiness/liveness probe와 resources requests/limits를 추가했다.
- `auth-service` Kubernetes 환경변수에 S3 endpoint, bucket, credential Secret 참조를 추가했다.

## OpenTelemetry + Prometheus/Grafana 관측성

- Spring Boot Actuator를 추가해 `/actuator/health`, `/actuator/prometheus` endpoint를 열었다.
- Micrometer Prometheus registry를 추가해 JVM, HTTP, Spring metrics를 Prometheus format으로 노출한다.
- Micrometer Tracing + OpenTelemetry OTLP exporter를 추가했다.
  - IDE 단독 실행은 collector가 없을 수 있어 tracing sampling 기본값을 `0.0`으로 둔다.
  - Docker Compose와 Kubernetes에서는 `MANAGEMENT_TRACING_SAMPLING_PROBABILITY=1.0`, `MANAGEMENT_OTLP_TRACING_ENDPOINT=http://tempo:4318/v1/traces`를 주입한다.
- 채팅 도메인 커스텀 metrics를 추가했다.
  - Kafka publish/consume duration/total
  - Elasticsearch index duration/total
  - Elasticsearch/MongoDB search duration/total
  - WebSocket broadcast duration/total
- Docker Compose에 Prometheus, Tempo, Grafana를 추가했다.
  - Prometheus: `http://localhost:9090`
  - Grafana: `http://localhost:3000`
  - Tempo: `http://localhost:3200`
  - Grafana 기본 계정: `admin / admin`
- Kubernetes에 Prometheus, Tempo, Grafana 매니페스트를 추가했다.
  - Prometheus NodePort: `30909`
  - Grafana NodePort: `30300`
  - Tempo는 클러스터 내부 OTLP 수집 endpoint로 사용한다.
- Grafana datasource provisioning을 추가해 `Kafka Talk Prometheus`, `Kafka Talk Tempo`가 자동 등록되도록 했다.
