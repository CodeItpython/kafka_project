## Kafka Kubernetes 안정화

- Kafka KRaft 단일 노드 Pod가 CrashLoopBackOff로 반복 재시작되는 문제를 확인했다.
- 원인:
  - `kafka` Service가 broker 포트 9092만 노출하고 controller 포트 9093을 노출하지 않았다.
  - 단일 노드 Kafka를 RollingUpdate로 교체하면서 같은 `node.id=1` broker/controller가 잠깐 둘 이상 떠서 controller registration이 꼬였다.
  - Service가 Ready Pod만 endpoint로 잡으면 Kafka가 controller에 붙어야 Ready가 되는 순환 대기가 발생한다.
- 조치:
  - `kafka` Service에 controller 포트 9093을 추가했다.
  - Kafka Deployment 전략을 `Recreate`로 변경했다.
  - Service에 `publishNotReadyAddresses: true`를 추가했다.
  - Kafka Pod에 readiness/liveness probe와 resources requests/limits를 추가했다.
- 결과:
  - `scripts/deploy-local-k8s.sh`로 이미지 빌드, `kubectl apply`, auth/frontend/discovery rollout을 검증했다.
  - Kafka 수정 후 `kubectl -n kafka-project get pods` 기준 전체 Pod가 Running 상태가 됐다.

## 사용자 프로필과 상태메시지

- `users` 테이블에 현재 프로필 상태를 표현하는 필드를 추가했다.
  - `statusMessage`: 사용자의 현재 상태메시지
  - `profileImageUrl`: 현재 프로필 이미지 URL
  - `updatedAt`: 프로필 마지막 수정 시각

  - 이름, 상태메시지, 프로필 이미지 URL, 이벤트 타입, 변경 시각을 남긴다.
  - 다른 사용자가 공개 프로필 화면에서 최근 변경 이력을 볼 수 있다.
- API를 추가했다.
  - `GET /api/users/me/profile`
  - `PATCH /api/users/me/profile`
  - `POST /api/users/me/profile-image`
  - `GET /api/users/{userId}/profile`
  - `GET /api/users/profile-images/{fileName}`
- 친구 목록 Redis 프로필 캐시에 상태메시지와 프로필 이미지 URL을 포함했다.
- 프론트엔드 사이드바에서 내 프로필 수정, 프로필 이미지 등록, 친구 공개 프로필/히스토리 조회, 공개 프로필에서 1:1 메시지 시작을 구현했다.
- 기존 PostgreSQL `users` 행이 있는 상태에서 `updated_at` 새 컬럼이 `NOT NULL`로 추가되면 애플리케이션 시작이 실패할 수 있어 nullable로 열고 애플리케이션 시작 후 백필하도록 보정했다.

## Jenkins CD와 채팅 화면 비율 보정

- Jenkins CD가 새 이미지를 만들어도 Kubernetes가 기존 `:local` 이미지를 계속 재사용할 수 있는 문제를 확인했다.
  - 원인: Kubernetes manifest는 `kafka-auth-service:local` 같은 고정 태그와 `imagePullPolicy: IfNotPresent`를 사용했다.
  - 조치: `scripts/docker-build.sh`, `scripts/deploy-local-k8s.sh`, `Jenkinsfile.local-deploy`에서 Git commit 기반 이미지 태그를 만들고 `kubectl set image`로 Deployment 이미지를 명시적으로 교체하도록 바꿨다.
  - 효과: 같은 `local` 태그 재사용으로 생기는 CD 반영 누락을 줄이고, 어떤 커밋 이미지가 배포됐는지 Kubernetes Deployment에서 추적할 수 있다.
- Kubernetes의 신규 `auth-service` Pod가 `/actuator/health/liveness`, `/actuator/health/readiness` probe에서 403을 받아 CrashLoopBackOff 되는 문제를 확인했다.
  - 조치: Spring Security에서 Actuator `health`, `prometheus` endpoint를 명시적으로 허용했다.
  - CD로 새 이미지가 배포되면 readiness/liveness probe가 인증 때문에 막히지 않는다.
- Testcontainers 통합 테스트를 CD 경로에 넣는 실험을 진행했으나, 현재 로컬 Gradle/Testcontainers 조합에서 `ImageNameSubstitutor` ServiceLoader 접근 오류가 발생했다.
  - 배포 파이프라인을 막지 않기 위해 `Jenkinsfile.local-deploy`와 `scripts/deploy-local-k8s.sh`는 다시 `./gradlew test` 기준으로 정리했다.
  - Testcontainers는 별도 브랜치에서 Gradle/JDK/Testcontainers 호환성을 먼저 고정한 뒤 재도입한다.
- 채팅방 진입 시 상단 대화 상대 헤더가 스크롤 위로 밀려 보이지 않는 UI 문제를 수정했다.
  - 채팅 shell을 viewport 높이 안에서 고정하고, 채팅 목록/대화 영역만 내부 스크롤되도록 정리했다.
  - 대화방 헤더는 `position: sticky`로 고정했다.
  - 방 진입 시 `window.scrollTo(0, 0)`을 호출해 이전 화면 스크롤 위치가 대화 화면에 남지 않게 했다.

## GraalVM Native Image 선택형 적용

- 현재 데이터 저장소 역할을 정리했다.
  - PostgreSQL: 유저, 이메일 인증, 프로필 히스토리, 채팅방, 채팅방 참여자/숨김 사용자
  - MongoDB: 채팅 메시지 원본, 첨부 메타데이터, 메시지 삭제 상태
  - Elasticsearch: MongoDB 메시지 검색용 색인
  - Redis: 프로필 캐시, unread count, 접속/typing 같은 짧은 상태
  - MinIO/S3: 이미지/GIF/프로필 이미지 파일 본문
- GraalVM Native Image는 `auth-service`에 선택형으로 적용했다.
  - Spring Boot 3의 AOT 처리와 GraalVM Native Build Tools를 사용한다.
  - 기본 Jenkins/CD는 안정성을 위해 JVM jar 이미지를 계속 사용한다.
  - `AUTH_IMAGE_MODE=native IMAGE_TAG=native-local scripts/docker-build.sh`를 실행하면 auth-service만 Native Image Dockerfile로 빌드한다.
- 검증 결과:
  - `./gradlew :auth-service:tasks --group build` 성공. `nativeCompile`, `nativeRun`, `collectReachabilityMetadata` task가 등록된 것을 확인했다.
  - `./gradlew :auth-service:test` 성공. 현재 테스트 소스는 없어 `NO-SOURCE`였지만 JVM 컴파일 경로는 통과했다.
  - `./gradlew :auth-service:nativeCompile` 성공. 첫 빌드 기준 약 12분 40초, 결과 실행 파일은 약 328MB다.
- `discovery-service`는 우선 JVM으로 유지했다.
  - Netflix Eureka 서버는 학습용 인프라 성격이 강하고, Native Image 호환성 검증보다 사용자 트래픽을 받는 `auth-service` 최적화가 먼저다.
