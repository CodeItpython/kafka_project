# 2026-06-21 작업 로그

## Testcontainers 통합 테스트 도입

- `auth-service` 테스트 의존성에 Spring Boot Testcontainers와 Testcontainers JUnit Jupiter 모듈을 추가했다.
- Spring Boot 3.3.6 기본 Testcontainers 1.19.8 대신 `testcontainers.version=1.21.3`을 사용하도록 조정했다.
- PostgreSQL, MongoDB, Elasticsearch, Redis, Kafka를 테스트 실행 시 실제 컨테이너로 띄우는 통합 테스트를 추가했다.
- 테스트는 JPA 사용자 저장, MongoDB 메시지 저장, Elasticsearch 검색 문서 저장, Redis key/value 저장, Kafka 메시지 발행을 검증한다.
- 외부 Eureka, OpenAI, Tempo, S3 의존성은 테스트에서 사용하지 않도록 속성을 고정했다.
- Docker Desktop 29에서 docker-java 기본 API 버전 `1.32`로 `/info`를 호출하면 빈 Docker 정보가 반환되어 Testcontainers가 Docker를 감지하지 못했다. 테스트 JVM에 `api.version=1.54`를 명시해 현재 Docker daemon API와 맞췄다.
- 로컬 macOS에서는 실제 Docker Desktop socket인 `~/.docker/run/docker.sock`을 우선 사용하고, Jenkins/Linux 환경에서는 `/var/run/docker.sock`을 fallback으로 사용한다.
- 검증 명령 `./gradlew :auth-service:cleanTest :auth-service:test --tests com.kafka.auth.infra.InfrastructureIntegrationTest --info`를 실행했고, 결과는 `tests=1, skipped=0, failures=0, errors=0`이다.

## 적용 개념

- Testcontainers: 테스트 코드가 Docker 컨테이너를 직접 띄워 실제 인프라와 거의 같은 조건에서 통합 테스트를 수행하는 도구다.
- DynamicPropertySource: 컨테이너가 실행된 뒤 할당된 포트와 접속 URL을 Spring Boot 테스트 환경 속성으로 주입한다.
- disabledWithoutDocker: Docker가 없는 환경에서는 통합 테스트를 실패시키지 않고 비활성화한다. Jenkins 로컬 CD 환경처럼 Docker가 있는 곳에서는 실제로 실행된다.
- docker-java API version: Testcontainers가 Docker daemon과 통신할 때 사용하는 REST API 버전이다. Docker Desktop이 너무 낮은 API 버전에 대해 호환 응답을 제대로 주지 않으면 `api.version`을 현재 daemon API에 맞춰 고정해야 한다.

## Debezium Outbox Pattern 도입

- 기존 채팅 메시지 전송 흐름은 API 요청 중 `KafkaTemplate.send()`를 바로 호출했다.
- 변경 후에는 `ChatService.publishMessage()`가 PostgreSQL `outbox_events` 테이블에 `ChatMessageCreated` 이벤트를 먼저 저장한다.
- `OutboxRelayScheduler`가 주기적으로 `PENDING`, `FAILED` outbox row를 조회하고 Kafka `chat-messages` 토픽으로 발행한다.
- 발행 성공 시 outbox 상태를 `PUBLISHED`로 바꾸고, 실패 시 `FAILED`로 남긴 뒤 backoff 후 재시도한다.
- 최대 재시도 횟수를 넘으면 `DEAD` 상태로 남겨 운영자가 확인할 수 있게 했다.
- Kafka 중복 발행 가능성에 대비해 consumer는 같은 `messageId`가 이미 MongoDB에 저장돼 있으면 unread count를 다시 올리지 않고 중복 이벤트를 건너뛴다.
- outbox 테이블은 Debezium Outbox Event Router 기본 컬럼명인 `id`, `aggregatetype`, `aggregateid`, `type`, `payload`, `timestamp`를 사용한다.
- 로컬 개발은 앱 내부 relay를 사용하고, 추후 Debezium Connect를 붙일 때는 `APP_OUTBOX_RELAY_ENABLED=false`로 앱 내부 relay를 끄면 된다.

## Outbox 적용 개념

- Transactional Outbox: 비즈니스 DB 변경과 이벤트 저장을 같은 트랜잭션으로 묶고, 별도 relay나 Debezium CDC가 이벤트를 Kafka로 발행하는 패턴이다.
- Debezium Outbox Event Router: PostgreSQL WAL을 읽어 outbox row를 Kafka 이벤트로 변환하는 Debezium SMT다. 앱은 Kafka 발행 책임에서 멀어지고 DB commit이 이벤트 발행의 기준점이 된다.
- Idempotency: outbox relay나 Kafka consumer 재시도 때문에 같은 이벤트가 두 번 처리될 수 있으므로 `messageId` 같은 고유 키로 중복 처리를 막아야 한다.

## FCM Push 알림 도메인 도입

- `notifications` 테이블을 추가해 채팅 메시지 수신자별 알림을 PostgreSQL에 저장하도록 했다.
- `push_device_tokens` 테이블을 추가해 브라우저/모바일 Push token을 사용자별로 관리할 수 있게 했다.
- `NotificationController`에 알림 목록 조회, WebSocket 구독 topic 조회, 전체 읽음, Push token 등록/해제 API를 추가했다.
- Kafka consumer가 메시지를 MongoDB에 저장하고 WebSocket으로 채팅 메시지를 broadcast한 뒤, `NotificationService`가 수신자별 알림을 만들도록 연결했다.
- 접속 중인 프론트엔드는 `/topic/notifications/{userHash}` topic을 구독해 실시간 알림을 받는다.
- Firebase Admin SDK를 추가했지만 `APP_PUSH_FCM_ENABLED=false`가 기본값이라, 로컬 키가 없어도 앱 실행과 테스트가 깨지지 않는다.
- Firebase 설정값이 있는 브라우저에서는 Firebase Web SDK로 FCM token을 발급해 `/api/notifications/push-tokens`에 등록한다.

## FCM 적용 개념

- FCM: Firebase Cloud Messaging. 앱 서버가 Firebase를 통해 웹/모바일 기기에 Push 알림을 전달하는 서비스다.
- Web Push token: 브라우저와 Firebase 프로젝트, VAPID key 조합으로 발급되는 기기 식별 토큰이다. 서버는 이 토큰으로 특정 브라우저에 알림을 보낸다.
- Foreground 알림과 Offline Push 분리: 접속 중에는 WebSocket이 빠르고, 접속하지 않았을 때는 FCM Push가 필요하다. 그래서 DB 알림 저장을 공통 기준으로 두고 WebSocket/FCM을 각각 전달 채널로 사용한다.
