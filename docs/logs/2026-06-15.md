## Jenkins main 기준 CI/CD 확인

- Jenkins 컨테이너 `kafka-jenkins`가 실행 중인 것을 확인했다.
- Jenkins job 설정을 컨테이너 내부 `config.xml` 기준으로 확인했다.
  - `kafka-chat-ci`: `*/main`, `Jenkinsfile`, Poll SCM `* * * * *`
  - `kafka-chat-main-ci`: `*/main`, `Jenkinsfile`, Poll SCM `* * * * *`
  - `kafka-chat-local-deploy`: `*/main`, `Jenkinsfile.local-deploy`, upstream `kafka-chat-main-ci`
- 최근 빌드 결과를 확인했다.
  - `kafka-chat-main-ci`: SUCCESS
  - `kafka-chat-local-deploy`: FAILURE
- CD 실패 원인:
  - Jenkins가 아직 GitHub `main`의 예전 `Jenkinsfile.local-deploy`를 사용하고 있었다.
  - 예전 파이프라인은 `rollout restart` 방식이라 같은 `:local` 이미지 태그 재사용 문제가 있었다.
  - auth-service 신규 Pod는 readiness/liveness probe에서 실패해 CrashLoopBackOff 상태였다.
- 조치:
  - `Jenkinsfile.local-deploy`의 upstream을 `kafka-chat-main-ci`로 명확히 맞췄다.
  - Docker 이미지를 Git commit 태그로 빌드하도록 바꿨다.
  - `kubectl rollout restart` 대신 `kubectl set image`로 Deployment 이미지를 명시적으로 교체하게 했다.
  - Spring Security에서 Actuator `health`, `prometheus` endpoint를 허용해 Kubernetes probe가 403으로 막히지 않게 했다.
- GitHub PR:
  - 브랜치 `codex/fix-elasticsearch-startup`를 원격에 push했다.
  - PR `https://github.com/CodeItpython/kafka_project/pull/7`을 생성했다.
  - Jenkins job은 현재 `main`만 감시하므로 PR 브랜치 push 자체로는 CD가 실행되지 않는다.
  - PR이 `main`에 병합되면 `kafka-chat-main-ci`가 main 변경을 감지하고, 성공 후 `kafka-chat-local-deploy`가 실행된다.

## Kubernetes와 컨테이너 구조 정리

- 현재 Kubernetes는 Docker Desktop의 로컬 단일 노드 클러스터에서 동작한다.
- Kafka 컨테이너 안에 Kubernetes가 들어간 구조가 아니라, Kubernetes가 Kafka/PostgreSQL/MongoDB/Redis/Elasticsearch/Spring Boot/React 등 여러 컨테이너를 Pod 단위로 관리한다.
- 현재 주요 Pod 역할:
  - `auth-service`: Spring Boot API, 인증, WebSocket, Kafka producer/consumer
  - `frontend`: React/Nginx 웹 UI
  - `discovery-service`: Eureka
  - `kafka`: 메시지 브로커
  - `postgres`: 유저, 이메일 인증, 프로필, 채팅방 데이터
  - `mongodb`: 채팅 메시지 원본
  - `redis`: 캐시, unread, online, typing 상태
  - `elasticsearch`: 채팅 검색 색인과 로그 저장소
  - `logstash`: JSON 로그 수집
  - `kibana`: Elasticsearch 조회 UI
  - `minio`: 이미지/GIF/파일 객체 저장
  - `prometheus`: metrics 수집
  - `tempo`: OpenTelemetry trace 저장
  - `grafana`: metrics/trace 시각화
- 현재 구성은 로컬 학습과 기능 검증에는 적절하다.
- 운영형으로 발전시키려면 PostgreSQL, MongoDB, Redis, Elasticsearch, Kafka 같은 상태 저장 컴포넌트는 StatefulSet, PVC, Operator 또는 managed service를 검토하는 것이 좋다.

## ELK, Prometheus, Tempo, Grafana 역할

- ELK:
  - Elasticsearch는 채팅 검색 색인과 Logstash 로그 인덱스를 저장한다.
  - Logstash는 auth-service의 JSON 로그를 받아 Elasticsearch에 적재한다.
  - Kibana는 Elasticsearch의 로그와 색인을 조회하는 UI다.
- Prometheus:
  - auth-service의 `/actuator/prometheus` endpoint를 10초마다 수집한다.
  - Kafka publish/consume, Elasticsearch index/search, WebSocket broadcast 같은 커스텀 metrics를 확인할 수 있다.
- Tempo:
  - auth-service가 OTLP로 보낸 trace를 저장한다.
  - 요청이 어떤 구간에서 느린지 추적하는 데 사용한다.
- Grafana:
  - Prometheus metrics와 Tempo trace를 한 화면에서 조회한다.
  - datasource는 `Kafka Talk Prometheus`, `Kafka Talk Tempo`로 자동 등록된다.
- 사용법과 접속 주소를 `docs/observability.md`와 `README.md`에 정리했다.

## OpenAI GPT 대화 요약

- 채팅방의 `GPT 요약` 버튼을 통해 최근 대화 내용을 요약하는 기능을 구성했다.
- 흐름:
  - 프론트엔드가 `POST /api/chat/rooms/{roomId}/summary` 호출
  - 백엔드가 채팅방 접근 권한 확인
  - MongoDB `chat_messages`에서 최근 메시지 조회
  - 삭제된 메시지와 빈 메시지를 제외
  - OpenAI Responses API로 요약 요청
  - 프론트엔드가 `대화 요약` 카드로 결과 표시
- 설정:
  - `OPENAI_API_KEY`
  - `OPENAI_MODEL`
  - `OPENAI_RESPONSES_URL`
- Docker Compose, Kubernetes, `.env.example`, `k8s/secrets.example.yaml`에 OpenAI 설정 주입 경로를 추가했다.
- ChatGPT Pro 구독과 OpenAI API Platform billing은 별도라는 점을 문서화했다.
- 자세한 사용법은 `docs/openai-summary.md`에 정리했다.

## GraalVM Native Image 적용

- GraalVM은 프로그래밍 언어가 아니라 Java 애플리케이션을 JVM 없이 실행 가능한 native executable로 컴파일할 수 있는 JVM/컴파일러 기술이다.
- 현재 프로젝트에서는 `auth-service`에 선택형으로 적용했다.
- 기본 CD는 안정성을 위해 JVM jar 이미지를 유지한다.
- 선택형 빌드:
  - `AUTH_IMAGE_MODE=native IMAGE_TAG=native-local scripts/docker-build.sh`
- 검증:
  - `./gradlew :auth-service:tasks --group build` 성공
  - `./gradlew :auth-service:test` 성공
  - `./gradlew :auth-service:nativeCompile` 성공
- 첫 native build는 약 12분 40초 걸렸고, 생성 파일은 약 313MB였다.
- 자세한 사용법은 `docs/graalvm-native.md`에 정리했다.
