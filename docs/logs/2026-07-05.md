## 개념

- HS256 키 길이: HMAC-SHA256은 최소 256비트(32바이트) 키가 필요하다. 기본/약한 시크릿이 프로덕션에 그대로 나가면 누구나 ADMIN 토큰을 위조할 수 있으므로, 기동 시 길이를 강제하고 개발 기본값 사용은 경고한다. (이 코드베이스의 JWT 검증은 헤더 alg를 신뢰하지 않고 항상 HMAC-SHA256으로 재계산·상수시간 비교하므로 alg-confusion 위험은 없다 — 문제는 시크릿 관리였다.)
- 원자적 카운터: `INCR` 후 별도 `EXPIRE`는 두 번의 왕복이라 사이에 장애가 나면 TTL 없는 키가 남아 버킷이 영구히 막힌다. Lua로 두 명령을 한 번에 실행하면 카운터 생성과 TTL 설정이 원자적으로 보장된다.
- Fail-open vs fail-closed: 가용성 우선 경로(일반 API)는 레이트리밋 백엔드 장애 시 요청을 통과시키고, 브루트포스 방어가 핵심인 인증 쓰기 경로는 차단(fail-closed)한다. 둘의 트레이드오프가 다르므로 정책을 엔드포인트 종류로 나눈다.
- X-Forwarded-For 신뢰: 클라이언트가 auth-service를 직접 호출하는 토폴로지에서는 XFF가 전적으로 공격자 통제 하에 있어, 값만 바꾸면 매 요청 새 버킷을 얻어 레이트리밋을 우회한다. 신뢰 프록시가 앞에 있을 때만 신뢰하도록 옵션화하고 기본은 원격 주소를 쓴다.
- SKIP LOCKED 아웃박스 릴레이: `SELECT ... FOR UPDATE SKIP LOCKED`는 다른 트랜잭션이 잠근 행을 건너뛰고 잠기지 않은 행만 가져온다(Hibernate lock-timeout 특수값 -2). 이벤트별로 짧은 트랜잭션에서 잠그고 전송하면, 여러 릴레이 워커가 겹치지 않는 이벤트를 동시에 처리할 수 있고 배치 전체 롤백으로 인한 중복 발행도 사라진다. 소비자는 message id 기준 멱등 처리를 이미 하므로 at-least-once로 안전하다.
