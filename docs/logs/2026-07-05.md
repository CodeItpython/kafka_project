## 첨부 서명 URL 설계·검증

`<img src>`는 Authorization 헤더를 못 실어 JWT 대신 HMAC 서명+만료를 URL에 붙였다(`?exp&sig`). 계층 흐름(엔티티→캐시 스냅샷→응답)에서 이중 서명·캐시 오염을 피하려 **응답 DTO 구성 계층에서만 서명**하고 캐시/DB에는 raw 경로를 저장한다. 첨부 URL은 클라이언트가 왕복 전송하므로 `publishMessage`에서 저장 전 `rawPath`로 정규화한다.

앱을 실제 기동(bootRun + 로컬 스토리지 + docker compose 인프라)해 실 HTTP로 검증(5/5 통과):

- 프로필/첨부 업로드 응답 URL에 `exp`/`sig` 포함
- 서명 URL GET → 200
- 무서명 URL GET → 403
- 변조 서명 GET → 403 (서명이 경로에 바인딩되어 다른 리소스/변조는 거부)

주의: 서명 TTL(기본 12h)은 방/프로필 Redis 캐시 TTL(30s/5분)보다 길어야 캐시된 서명이 유효하다. `app.storage.signing.secret`/`app.storage.signing.ttl`로 조정한다.

## 개념

- STOMP 채널 인터셉터 인증: WebSocket 핸드셰이크(HTTP 업그레이드)는 열어두고, STOMP CONNECT 프레임의 네이티브 헤더로 JWT를 받아 인증한다. 인바운드 채널 인터셉터에서 `MessageHeaderAccessor.getAccessor`(wrap 아님 — wrap은 복사본이라 setUser가 세션에 전파되지 않음)로 가변 접근자를 얻어 CONNECT에서 `setUser`하면 이후 SUBSCRIBE 프레임에서 `getUser()`로 주체를 얻는다.
- SUBSCRIBE 인가: 브로드캐스트 토픽(`/topic/rooms/{id}`, `/topic/notifications/{hash}`)은 아무나 구독하면 도청이 되므로, 방 토픽은 방 접근 권한을, 알림 토픽은 sha256(본인 이메일) 일치를 검증한다. 접근 판정은 lazy 컬렉션을 읽으므로 트랜잭션 안에서 수행한다.
- 6자리 코드: 인증코드 공간을 10^4→10^6으로 넓혀 코드당 브루트포스 성공 확률을 100분의 1로 낮춘다. 코드 길이는 프론트 OTP UI와 백엔드 생성·검증 패턴이 함께 맞아야 하므로 양쪽을 동시에 변경한다.
