## 첨부 서명 URL 설계·검증

`<img src>`는 Authorization 헤더를 못 실어 JWT 대신 HMAC 서명+만료를 URL에 붙였다(`?exp&sig`). 계층 흐름(엔티티→캐시 스냅샷→응답)에서 이중 서명·캐시 오염을 피하려 **응답 DTO 구성 계층에서만 서명**하고 캐시/DB에는 raw 경로를 저장한다. 첨부 URL은 클라이언트가 왕복 전송하므로 `publishMessage`에서 저장 전 `rawPath`로 정규화한다.

앱을 실제 기동(bootRun + 로컬 스토리지 + docker compose 인프라)해 실 HTTP로 검증(5/5 통과): 업로드 응답 URL에 `exp`/`sig` 포함 / 서명 URL GET 200 / 무서명 GET 403 / 변조 서명 GET 403(서명이 경로에 바인딩). 주의: 서명 TTL(기본 12h)은 방/프로필 Redis 캐시 TTL(30s/5분)보다 길어야 캐시된 서명이 유효하다.

## 개념

- HS256 키 길이: HMAC-SHA256은 최소 256비트(32바이트) 키가 필요하다. 기본/약한 시크릿이 프로덕션에 그대로 나가면 누구나 ADMIN 토큰을 위조할 수 있으므로, 기동 시 길이를 강제하고 개발 기본값 사용은 경고한다. (이 코드베이스의 JWT 검증은 헤더 alg를 신뢰하지 않고 항상 HMAC-SHA256으로 재계산·상수시간 비교하므로 alg-confusion 위험은 없다 — 문제는 시크릿 관리였다.)
- 원자적 카운터: `INCR` 후 별도 `EXPIRE`는 두 번의 왕복이라 사이에 장애가 나면 TTL 없는 키가 남아 버킷이 영구히 막힌다. Lua로 두 명령을 한 번에 실행하면 카운터 생성과 TTL 설정이 원자적으로 보장된다.
- Fail-open vs fail-closed: 가용성 우선 경로(일반 API)는 레이트리밋 백엔드 장애 시 요청을 통과시키고, 브루트포스 방어가 핵심인 인증 쓰기 경로는 차단(fail-closed)한다.
- X-Forwarded-For 신뢰: 클라이언트가 auth-service를 직접 호출하는 토폴로지에서는 XFF가 전적으로 공격자 통제 하에 있어, 값만 바꾸면 매 요청 새 버킷을 얻어 레이트리밋을 우회한다. 신뢰 프록시가 앞에 있을 때만 신뢰하도록 옵션화하고 기본은 원격 주소를 쓴다.
- SKIP LOCKED 아웃박스 릴레이: `SELECT ... FOR UPDATE SKIP LOCKED`는 다른 트랜잭션이 잠근 행을 건너뛰고 잠기지 않은 행만 가져온다. 이벤트별 짧은 트랜잭션에서 잠그고 전송하면 여러 릴레이 워커가 겹치지 않는 이벤트를 동시에 처리한다. 소비자는 message id 기준 멱등이라 at-least-once로 안전하다.
- STOMP 채널 인터셉터 인증: WebSocket 핸드셰이크는 열어두고 STOMP CONNECT 프레임의 네이티브 헤더로 JWT를 받아 인증한다. 인바운드 채널 인터셉터에서 `MessageHeaderAccessor.getAccessor`(wrap 아님 — wrap은 복사본이라 setUser가 세션에 전파되지 않음)로 가변 접근자를 얻어 CONNECT에서 `setUser`하면 이후 SUBSCRIBE 프레임에서 `getUser()`로 주체를 얻는다.
- SUBSCRIBE 인가: 브로드캐스트 토픽(`/topic/rooms/{id}`, `/topic/notifications/{hash}`)은 아무나 구독하면 도청이 되므로, 방 토픽은 방 접근 권한을, 알림 토픽은 sha256(본인 이메일) 일치를 검증한다.
- 6자리 코드: 인증코드 공간을 10^4→10^6으로 넓혀 코드당 브루트포스 성공 확률을 100분의 1로 낮춘다. 프론트 OTP UI와 백엔드 생성·검증 패턴을 함께 변경해야 한다.
