## 개념 정리

- Elasticsearch의 색인어는 일반 DB row처럼 별도 테이블에서 조회하는 개념이 아니라 inverted index 내부 term으로 저장된다.
- analyzer가 텍스트를 어떤 token으로 나누는지는 `_analyze` API로 확인한다.
- 실제 특정 문서에 저장된 term 정보는 `_termvectors` API로 확인한다.
- 현재 메시지 색인 흐름은 `메시지 전송 -> Kafka publish -> Kafka consumer -> MongoDB 저장 -> Elasticsearch 저장` 순서다.
- 애플리케이션을 통해 들어온 새 메시지는 consumer가 처리하는 즉시 Elasticsearch에 색인된다.
- MongoDB에 직접 넣은 데이터는 자동으로 Elasticsearch에 들어가지 않으므로 별도 reindex 작업이 필요하다.
- Kubernetes는 백엔드를 실행하기 위한 필수 조건은 아니다. 로컬 개발은 Gradle bootRun으로 실행할 수 있고, Kubernetes는 Docker 이미지로 포장한 서비스를 운영 방식에 가깝게 배포할 때 사용한다.
