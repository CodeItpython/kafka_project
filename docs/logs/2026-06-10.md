## 개념 정리

- Elasticsearch의 색인어는 일반 DB row처럼 별도 테이블에서 조회하는 개념이 아니라 inverted index 내부 term으로 저장된다.
- analyzer가 텍스트를 어떤 token으로 나누는지는 `_analyze` API로 확인한다.
- 실제 특정 문서에 저장된 term 정보는 `_termvectors` API로 확인한다.
- 현재 메시지 색인 흐름은 `메시지 전송 -> Kafka publish -> Kafka consumer -> MongoDB 저장 -> Elasticsearch 저장` 순서다.
- 애플리케이션을 통해 들어온 새 메시지는 consumer가 처리하는 즉시 Elasticsearch에 색인된다.
- MongoDB에 직접 넣은 데이터는 자동으로 Elasticsearch에 들어가지 않으므로 별도 reindex 작업이 필요하다.
- Kubernetes는 백엔드를 실행하기 위한 필수 조건은 아니다. 로컬 개발은 Gradle bootRun으로 실행할 수 있고, Kubernetes는 Docker 이미지로 포장한 서비스를 운영 방식에 가깝게 배포할 때 사용한다.

## 추가 작업 - 이미지/GIF 첨부와 삭제 기능

- 채팅 메시지에 이미지/GIF 첨부를 보낼 수 있도록 multipart 업로드 API를 추가했다.
- 첨부 파일은 로컬 `uploads/chat` 경로에 UUID 파일명으로 저장하고, 메시지에는 URL, MIME type, 원본 파일명, 파일 크기만 저장한다.
- `<img>` 태그가 JWT 헤더를 붙일 수 없으므로 첨부 파일 조회 API `GET /api/chat/attachments/{fileName}`는 공개 조회로 열어두고, 파일명은 UUID로 추측하기 어렵게 처리했다.
- Kafka 메시지 이벤트에 첨부 메타데이터를 포함해서 producer와 consumer 사이에 이미지/GIF 메시지가 전달되도록 했다.
- MongoDB 채팅 메시지 문서에 첨부 정보, 모두에게 삭제 여부, 나에게만 삭제한 사용자 목록을 추가했다.
- “나에게 삭제”는 메시지 문서에 현재 사용자 이메일을 기록해 내 조회/검색 결과에서만 숨긴다.
- “모두에게 삭제”는 작성자만 실행할 수 있고, 원문과 첨부 정보를 제거한 뒤 WebSocket으로 삭제 상태를 브로드캐스트한다.
- 모두에게 삭제된 메시지는 Elasticsearch 색인에서도 제거해서 검색 결과에 남지 않게 했다.
- 선택한 채팅방의 대화내용을 내 화면에서 모두 비우는 API와 프론트 버튼을 추가했다.
- 채팅방 삭제는 물리 삭제가 아니라 현재 사용자에게만 방 목록에서 숨기는 방식으로 구현했다.
- React 채팅 UI에 첨부 버튼, 미리보기, 이미지/GIF 렌더링, 메시지별 삭제 버튼, 대화 비우기, 채팅방 삭제 버튼을 추가했다.
