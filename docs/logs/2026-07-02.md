## Kafka DLT replay API

- DLT에 격리된 `ChatMessageEvent`를 조회하고 원본 topic으로 재발행하는 관리자 API를 추가했다.
- `GET /api/admin/kafka/dlt/messages?limit=20`로 `chat-messages-dlt` 메시지를 partition, offset, messageId, roomId, senderEmail 기준으로 확인할 수 있게 했다.
- `POST /api/admin/kafka/dlt/replay`로 선택한 messageId 또는 제한 개수만큼 DLT 메시지를 원본 `chat-messages` topic에 다시 발행할 수 있게 했다.
- replay 요청은 `dryRun`을 지원해 실제 재발행 전에 대상 메시지를 먼저 확인할 수 있다.
- `app.chat.dlt.topic` 설정을 추가해 DLT topic 이름을 환경별로 바꿀 수 있게 했다.
- README에 DLT replay API 사용법과 주의사항을 정리했다.

## 사용한 개념

- DLT replay: Dead Letter Topic에 격리된 메시지를 장애 원인 수정 후 원본 topic으로 다시 발행해 정상 consumer 흐름으로 복구하는 운영 절차다.
- Dry run: 실제 재발행 없이 대상 메시지만 확인해 운영 실수를 줄이는 안전장치다.
- Manual Kafka consumer assign: replay 도구는 consumer group offset에 의존하지 않고 DLT partition을 직접 assign하고 처음 offset부터 읽는다.

## DLT 운영 패널

- 프론트엔드 친구 사이드바에 `실패 메시지` 운영 패널을 추가했다.
- 운영 패널에서 DLT 메시지를 조회하고 messageId 단위로 선택할 수 있게 했다.
- `미리 확인` 버튼은 `dryRun=true`로 replay 후보만 검증하고, `재처리` 버튼은 선택된 메시지를 원본 topic으로 재발행한다.
- 재처리 결과는 스캔 개수, 대상 개수, source topic, target topic으로 화면에 표시한다.
- `users.role` 기반 관리자 권한을 추가하고 `/api/admin/**`는 `ROLE_ADMIN` 사용자만 접근하게 제한했다.
- 개발 seed 계정 중 `user@example.com`을 관리자 계정으로 지정해 DLT 운영 패널 테스트가 가능하게 했다.
- `admin_audit_events` 테이블과 `/api/admin/audit-events` 조회 API를 추가했다.
- DLT dry-run과 실제 replay 요청은 성공/실패 여부, 실행자, source topic, target topic, 대상 messageId, 오류 메시지를 감사 로그로 남기게 했다.

## 추가 사용 개념

- RBAC: 로그인 여부와 별개로 사용자 역할에 따라 관리자 API 접근을 제한하는 권한 제어 방식이다.
- 감사 로그: 운영자가 장애 복구 명령을 실행한 시점과 결과를 별도 테이블에 남겨 사고 추적과 재현을 돕는 기록이다.

## Flyway migration

- Flyway migration을 도입했다. `spring.jpa.hibernate.ddl-auto=update`를 `validate`로 바꾸고, PostgreSQL 기준 초기 스키마를 `V1__baseline_postgresql_schema.sql`에 정의했다.
- 새 DB는 Flyway가 `users`, 채팅방, 읽음/전달 상태, 알림, Push token, Outbox, 관리자 감사 로그 테이블을 순서대로 생성한다.
- 기존 로컬 DB는 `spring.flyway.baseline-on-migrate=true`로 Flyway 이력 테이블을 만들 수 있게 했다.
- 이미 만들어져 있던 로컬 DB가 version 1로 baseline 처리된 뒤 최신 감사 로그 테이블을 놓치지 않도록 `V2__admin_audit_events.sql` 보강 migration을 추가했다.

## 추가 사용 개념

- Flyway: DB 변경 이력을 `V1__...sql`, `V2__...sql` 같은 버전 파일로 관리하고 애플리케이션 시작 시 아직 적용되지 않은 변경만 순서대로 실행하는 migration 도구다.
- `ddl-auto=validate`: Hibernate가 테이블을 생성/수정하지 않고 엔티티 매핑과 실제 DB 스키마가 맞는지 검증만 수행하게 하는 설정이다.

## Redis API rate limit

- Redis 기반 API rate limit을 추가했다. `/api/auth/login`, 회원가입, 이메일 인증 코드 발급, 이메일 로그인, 카카오 OAuth 시작 API는 기본 1분 20회로 제한한다.
- 나머지 `/api/**` 요청은 기본 1분 600회 제한을 적용해 개발 중 사용성은 유지하면서 비정상적인 반복 호출을 제어한다.
- 제한에 걸리면 HTTP 429와 `Retry-After`, `X-RateLimit-*` 헤더를 내려주고, Redis 장애 시에는 로그인 자체가 막히지 않도록 fail-open으로 처리한다.

## 추가 사용 개념

- Rate limiting: 특정 시간 창 안에서 클라이언트가 호출할 수 있는 요청 수를 제한해 brute-force 로그인, 인증 코드 남발, API 과호출을 줄이는 보호 장치다.
- Fixed window counter: 시간 구간을 나누고 Redis key별 카운터를 증가시키는 단순하고 빠른 제한 방식이다.
