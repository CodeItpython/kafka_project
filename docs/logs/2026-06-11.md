## 확인된 이슈

- 로컬 `kubectl config get-contexts` 결과가 비어 있었다.
- Jenkins 컨테이너 내부에서도 `kubectl config current-context`가 설정되어 있지 않았다.
- 따라서 현재 상태에서는 Jenkins가 Docker 이미지를 빌드할 수는 있지만, 로컬 Kubernetes 배포 단계는 Docker Desktop Kubernetes context를 먼저 설정해야 성공한다.
- push 이후 Jenkins Poll SCM이 새 build를 생성하는 것을 확인했다.
- 첫 자동 build는 Jenkinsfile을 lightweight checkout으로 읽는 단계에서 `fatal: not in a git directory` 오류로 실패했다.
- 두 Jenkins job의 `lightweight` 설정을 `false`로 바꿔 전체 checkout 방식으로 Jenkinsfile을 읽도록 수정했다.
- Pipeline job에서는 Poll SCM 트리거가 `PipelineTriggersJobProperty` 아래에 있어야 하므로, `kafka-chat-ci`와 `kafka-chat-local-deploy` job XML을 해당 구조로 보정했다.
- Jenkins를 재시작해 수정된 job 설정을 다시 읽도록 했다.
- Jenkins 로컬 배포 모드는 Docker 접근까지 준비됐고, Kubernetes 배포는 로컬 K8s context 생성 후 검증해야 하는 상태로 정리했다.
- Jenkins job 설정을 컨테이너 내부 수동 XML 수정에만 의존하지 않도록 `jenkins/init.groovy.d/01-local-pipelines.groovy`를 추가했다.
- Jenkins 이미지가 시작될 때 `kafka-chat-ci`와 `kafka-chat-local-deploy` Pipeline job을 생성/보정하고, Poll SCM과 전체 checkout 설정을 자동 적용하도록 구성했다.
- 로컬 검증에서 Poll SCM 주기를 명확하게 보기 위해 Jenkinsfile과 Jenkins 초기화 스크립트의 주기를 `* * * * *`로 통일했다.
- 기존 Jenkins volume에도 최신 init 스크립트가 바로 반영되도록 `docker-compose.yml`에서 `./jenkins/init.groovy.d`를 `/var/jenkins_home/init.groovy.d`에 read-only mount했다.

## 사용된 개념

- OAuth 2.0 Authorization Code Flow: 외부 로그인 제공자에게 인증을 맡기고, 서버가 authorization code를 access token으로 교환한 뒤 내부 JWT를 발급하는 방식.
- Secret externalization: 민감정보를 `application.properties`에 직접 쓰지 않고 환경변수, `.env`, Jenkins Credentials, Kubernetes Secret 등 실행 환경에서 주입하는 방식.
- Responsive master-detail UI: 데스크톱에서는 목록과 상세를 같이 보여주고, 모바일에서는 목록과 상세를 화면 단위로 전환하는 패턴.
- Database check constraint: 컬럼에 들어갈 수 있는 값을 DB 레벨에서 제한하는 제약조건이다. 애플리케이션 enum에 값을 추가하면 운영 DB에서는 migration 도구로 같은 제약조건도 함께 변경해야 한다.
- Local schema patch: 로컬 개발 환경에서는 빠르게 개발을 이어가기 위해 기동 시점에 작은 스키마 보정을 수행할 수 있다. 운영 환경에서는 Flyway, Liquibase 같은 명시적 migration으로 관리하는 것이 더 적합하다.
- Elasticsearch autocomplete: 색인된 텍스트를 기반으로 사용자가 입력 중인 prefix 또는 포함 문자열에 맞는 후보를 빠르게 제안하는 기능이다. 운영 수준에서는 `completion suggester`, `search_as_you_type`, `edge_ngram` 매핑을 선택해 성능과 한글 형태소 품질을 조정할 수 있다.
- Debounce: 키 입력마다 즉시 요청하지 않고 짧은 대기 시간을 둔 뒤 마지막 입력만 서버에 보내는 방식이다. 자동완성 요청 폭주를 줄인다.
- Container image: Spring Boot jar 파일, JDK 런타임, 실행 명령, 필요한 메타데이터를 하나로 묶은 실행 패키지다. jar를 Docker 컨테이너에 직접 넣어 실행한다기보다, jar가 들어 있는 이미지를 만들고 그 이미지로 컨테이너를 실행한다.
- Kubernetes deployment: Kubernetes가 직접 jar를 이해해서 실행하는 것이 아니라, Deployment가 지정한 컨테이너 이미지를 가져와 Pod 안에서 컨테이너로 실행한다. Service는 그 Pod로 트래픽을 라우팅한다.
