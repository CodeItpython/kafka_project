# 2026-06-11 작업 로그

## 작업 내용

- 사용자가 Jenkins 로컬 배포 모드의 위험을 이해하고 승인했다.
- `docker-compose.jenkins-deploy.yml` 오버라이드로 Jenkins를 재기동했다.
- Jenkins 컨테이너에 Docker socket이 `/var/run/docker.sock`으로 mount된 것을 확인했다.
- Jenkins 컨테이너에 `~/.kube`가 `/var/jenkins_home/.kube`로 read-only mount된 것을 확인했다.
- Jenkins 내부에서 `docker ps`가 동작하는 것을 확인했다.
- `kafka-chat-ci`와 `kafka-chat-local-deploy` job 모두 `H/1 * * * *` Poll SCM 트리거를 유지하고 있는 것을 확인했다.

## 확인된 이슈

- 로컬 `kubectl config get-contexts` 결과가 비어 있었다.
- Jenkins 컨테이너 내부에서도 `kubectl config current-context`가 설정되어 있지 않았다.
- 따라서 현재 상태에서는 Jenkins가 Docker 이미지를 빌드할 수는 있지만, 로컬 Kubernetes 배포 단계는 Docker Desktop Kubernetes context를 먼저 설정해야 성공한다.
- push 이후 Jenkins Poll SCM이 새 build를 생성하는 것을 확인했다.
- 첫 자동 build는 Jenkinsfile을 lightweight checkout으로 읽는 단계에서 `fatal: not in a git directory` 오류로 실패했다.
- 두 Jenkins job의 `lightweight` 설정을 `false`로 바꿔 전체 checkout 방식으로 Jenkinsfile을 읽도록 수정했다.
- Pipeline job에서는 Poll SCM 트리거가 `PipelineTriggersJobProperty` 아래에 있어야 하므로, `kafka-chat-ci`와 `kafka-chat-local-deploy` job XML을 해당 구조로 보정했다.
- Jenkins를 재시작해 수정된 job 설정을 다시 읽도록 했다.
- Jenkins 로컬 배포 모드는 Docker 접근까지 준비됐고, Kubernetes 배포는 로컬 K8s context 생성 후 검증해야 하는 상태로 정리했다.
- Jenkins job 설정을 컨테이너 내부 수동 XML 수정에만 의존하지 않도록 `jenkins/init.groovy.d/01-local-pipelines.groovy`를 추가했다.
- Jenkins 이미지가 시작될 때 `kafka-chat-ci`와 `kafka-chat-local-deploy` Pipeline job을 생성/보정하고, Poll SCM과 전체 checkout 설정을 자동 적용하도록 구성했다.
- 로컬 검증에서 Poll SCM 주기를 명확하게 보기 위해 Jenkinsfile과 Jenkins 초기화 스크립트의 주기를 `* * * * *`로 통일했다.
- 기존 Jenkins volume에도 최신 init 스크립트가 바로 반영되도록 `docker-compose.yml`에서 `./jenkins/init.groovy.d`를 `/var/jenkins_home/init.groovy.d`에 read-only mount했다.
- Jenkins checkout workspace가 깨져 `fatal: not in a git directory`가 반복되는 문제를 줄이기 위해 두 Jenkinsfile에 `skipDefaultCheckout(true)`와 `deleteDir()` 기반 checkout을 적용했다.
- 로컬 배포 job은 CI 성공 후 이어서 실행되도록 upstream trigger를 추가했다.
- push 후 `kafka-chat-ci #4`가 자동 실행되어 backend Gradle test와 frontend build가 성공하는 것을 확인했다.
- `kafka-chat-ci #4` 성공 후 `kafka-chat-local-deploy #2`가 upstream trigger로 자동 실행되는 것을 확인했다.
- `kafka-chat-local-deploy #2`는 Docker 이미지 `kafka-discovery-service:local`, `kafka-auth-service:local`, `kafka-frontend:local` 빌드까지 성공했다.
- `kubectl apply -k k8s` 단계는 로컬 Kubernetes context가 설정되어 있지 않아 실패했다. Docker Desktop Kubernetes를 켜고 `kubectl config current-context`가 정상 context를 반환하도록 한 뒤 재실행해야 한다.

## 다음 조치

- Docker Desktop 설정에서 Kubernetes를 활성화한다.
- 터미널에서 `kubectl config get-contexts`와 `kubectl config current-context`가 정상 출력되는지 확인한다.
- context가 정상화되면 `kafka-chat-local-deploy` job을 실행해 K8s 배포까지 검증한다.
