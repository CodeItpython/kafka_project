## 사용한 개념

- Explicit room membership: 채팅방 접근, presence, unread, 알림 대상을 `chat_room_participants` 기준으로 계산한다.
- Backward compatibility: 기존 데이터 중 참여자 목록이 비어 있는 그룹방은 전체 공개방으로 해석해 갑작스러운 접근 차단을 피한다.
- Progressive enhancement: 방 목록에 `participantCount`를 포함해 프론트엔드가 별도 조회 없이 목록에서 참여자 수를 표시할 수 있게 한다.

## 브라우저 알림 표시 개선

- STOMP 실시간 알림을 수신했을 때 앱 내부 알림 목록만 갱신하던 흐름에 브라우저 Notification API 표시를 추가했다.
- Firebase/FCM 환경변수가 없어도 로컬 개발에서는 로그인 후 알림 권한만 허용하면 새 메시지 알림이 OS 브라우저 알림으로 표시된다.
- FCM 설정이 있는 경우에는 기존 서비스워커 기반 백그라운드 푸시 토큰 등록을 유지한다.
- 백그라운드 푸시 알림을 클릭해 `/?roomId=...`로 앱이 열리면 해당 채팅방을 자동으로 열도록 딥링크 처리를 추가했다.

## 사용한 개념

- Notification API: 브라우저가 제공하는 로컬 알림 기능으로, 권한이 `granted`일 때 `new Notification(...)`을 호출해 OS 알림을 띄운다.
- Progressive fallback: FCM 설정이 없거나 FCM 등록에 실패해도 WebSocket/STOMP 실시간 이벤트를 받은 열린 브라우저 탭에서는 알림을 표시한다.
- Notification deep link: 알림 클릭 시 대상 채팅방 ID를 URL 쿼리와 앱 상태로 연결해 바로 대화방으로 이동한다.

## 메시지 전달 상태 ACK

- 메시지별 수신자 전달 상태를 저장하는 `chat_message_delivery_states` JPA 엔티티와 Repository를 추가했다.
- Kafka consumer가 메시지를 MongoDB에 저장한 뒤 수신자별 `SENT` 상태를 생성하도록 연결했다.
- 수신자 브라우저가 알림 이벤트를 받으면 `POST /api/chat/rooms/{roomId}/messages/{messageId}/delivered`를 호출해 `DELIVERED` 상태로 승격한다.
- 채팅방 읽음 처리 시 기존 전달 상태도 `READ`로 승격해 `SENT -> DELIVERED -> READ` 흐름이 유지되게 했다.
- `/topic/rooms/{roomId}/delivery-states` WebSocket topic을 추가해 보낸 사람 화면의 `전송됨`, `전달됨`, `읽음` 표시가 실시간으로 갱신되게 했다.
- 통합 테스트에 전달 ACK와 읽음 승격 검증을 추가했다.

## 사용한 개념

- Delivery ACK: 수신 클라이언트가 메시지를 실제로 받았음을 서버에 확인해 주는 명시적 신호다.
- Per-recipient message state: 같은 메시지라도 사용자마다 상태가 다르므로 메시지 본문과 분리된 사용자별 상태 테이블로 관리한다.
- Status precedence: `READ`가 가장 강한 상태이고, 그다음 `DELIVERED`, 마지막이 `SENT`다.

## Kafka Retry Topic / DLT

- `ChatService.persistAndBroadcast` Kafka consumer에 Spring Kafka `@RetryableTopic`을 적용했다.
- 기본 재시도 정책은 총 3회 시도, 2초 backoff, 2.0 multiplier로 설정했다.
- 실패한 채팅 메시지 이벤트는 `chat-messages` 기반 retry topic으로 이동하고, 계속 실패하면 DLT suffix topic으로 격리된다.
- `@DltHandler`를 추가해 최종 실패 이벤트의 `messageId`, `roomId`, `senderEmail`을 error log로 남기도록 했다.
- DLT 이동 건수를 Prometheus에서 볼 수 있도록 `kafka.talk.kafka.dlt.total` counter를 추가했다.
- README에 retry/DLT 흐름과 topic 확인 명령을 정리했다.

## 사용한 개념

- Retry Topic: consumer 처리 실패 시 같은 consumer thread에서 무한 재시도하지 않고 별도 topic으로 옮겨 backoff 후 다시 처리하는 패턴이다.
- DLT: Dead Letter Topic의 약자로, 정해진 재시도 횟수 이후에도 실패한 메시지를 유실하지 않고 격리하는 topic이다.
- Observability: DLT 이동을 metric과 log로 남겨 장애 원인과 복구 대상을 추적한다.
